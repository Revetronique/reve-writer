# reve-writer
(作成者) Revetronique
(作成日) 2026/6/17

## 概要

運用しているWordPressブログの記事自動生成システム。
reve-writer（Claude.ai）→ Notion → Claude Code → WordPress のパイプライン。
カスタムコードは使わない。Notion + Claude Code設定 + MCP で完結。

- **ワークフロー手順** は `.claude/skills/` の各スキルに定義（下記スキル一覧）
- **実装コード（アップロード・投稿等）** は `.claude/agents/` の各エージェント定義が正
- **システム設計・DBスキーマ全体** は `docs/design/reve-writer-design.md` を参照
- **WP投稿のトラブル対応** は `docs/design/wp-post-troubleshooting.md` を参照

## 共通制約（全作業で必ず守る）

- WordPress投稿は**必ず draft**。公開は手動。`"status": "publish"` は絶対に送らない。
- 写真のバイナリデータを Claude API に送信しない（NAS→WP直接送信、解析はローカルOllama）。
- NASアクセスは SMB マウント（LAN内のみ）。`ls` によるファイル名取得のみ行う。
- アップロード・投稿の前にユーザー確認を必ず挟む（複数記事でも確認画面は1回にまとめる）。
- アニメのスクリーンショットは使用しない（著作権リスク）。
- 記事本文は2,000〜2,500文字を目安（SiteGuard LiteのPOSTボディサイズ制限のため）。
- Notion下書きページの取得は必ず `notion-fetch` でURLを直接指定する。
  **`notion-search` は使用禁止**（全文検索になり低速のため）。

| やりたいこと | スキル |
|---|---|
| Notion下書きをWordPressへ投稿（写真マッチング・校正・確認・投稿・ステータス更新） | `/post-draft` |

| NAS写真の一括リネーム | `/rename-photos` |
| 公開前チェック（9項目の自動確認） | `/precheck` |
| 既存投稿の修正（セクション書き直し・メタ・タイトル/スラッグ） | `/edit-post` |

受け取ったら `/post-draft` を実行する。複数ブロックがある場合も1回の `/post-draft` で並列処理する。

## サブエージェント構成

定義・実装コードは `.claude/agents/*.md` が唯一の正。

| エージェント | 役割 | フェーズ |
|---|---|---|
| `image-indexer` | image_indexer.pyを実行してmanifest.jsonを生成 | フェーズ-1（任意・事前準備） |
| `image-selector` | manifest.json＋下書きから挿入写真を選定 | フェーズ0（任意・manifest存在時のみ） |
| `photo-uploader` | NAS→WP Media APIへ並列アップロード | フェーズ1（並列） |
| `proofreader` | 文体・スタイルガイド準拠チェック | フェーズ1（並列） |
| `wp-poster` | WordPress REST API投稿 | フェーズ2（ユーザー確認後のみ） |

- フェーズ1（photo-uploader / proofreader）は記事をまたいで全て同時起動する。直列処理は禁止。
- wp-poster はユーザー確認の完了後のみ実行する。

## Notion DB構成

| DB | 役割 | 主な用途 |
|---|---|---|
| ラブライブ作品別DB | 作品マスター | ラブライブシリーズの作品情報 |
| 聖地スポットDB | スポットマスター | 聖地の位置・情報（座標はWGS84 小数点6桁） |
| 登場シーンDB | 登場情報 | スポット × 話数/MV（1登場=1レコード） |
| ご当地コラボDB | コラボ情報 | 聖地巡礼先の現地コラボ情報 |
| 来訪記録DB（Visits） | 巡礼ログ | 訪問記録（Status: Draft/Writing/Published） |
| Events DB | イベント・お出かけ | Type D/E の素材（ステータス: 予定〜公開済み） |
| 下書き | 記事下書き置き場 | reve-writerが生成した下書き。Claude Codeが取得してWPに投稿 |
| Claude Code 連携ルール | システム連携 | reve-writerとNotion、Claude Codeの連携ドキュメント |

- 下書きページのタイトル形式: `YYYY-MM-DD_{スラッグ}`。冒頭の `---` より前がメタ情報、後が本文。
- 下書きステータス遷移: `下書き生成済み`（reve-writer保存直後）→ `投稿済み`（Claude CodeがPOST後に更新）
- 写真関連プロパティ（Visits/Events共通）: `写真フォルダ`（NASパス）、`選定写真リスト`
  （ファイル名・id・urlのセット。例: `20240503-numazu-0001.jpg(id:1234|url:https://...)`）
- 各DBの全プロパティ定義は `docs/design/reve-writer-design.md` 参照

## 文章スタイル（全記事タイプ共通）

Notion の Writing Style ページも参照のこと。

### 基本ルール

- 冒頭: 「どうも、Reveです。」
- 締め: 「もし参考になれば幸いです。」
- 見出し: 【】形式（例: 【会場の様子】）
- 一人称: 「当方」、「我々」または省略
- 読者: 皆さん
- 語尾: です・ます調、会話的で丁寧だが堅すぎない
- 適度にカジュアル表現を混ぜる:（汗）、ｗ、「〜なんですよね」等
- 段落は3〜4文で改行（読みやすさ重視）
- 記事の長さ: 2,000〜2,500文字を目安

### 写真の挿入ルール

- 各セクションに1〜3枚、キャプション付き
- 選定写真リストのファイル名でNASパスと照合

### メモ → 文章の変換ルール

- メモの箇条書きをそのまま並べただけの文章にしない
- 「〜とのこと」「〜だそうです」で伝聞と体験を区別する
- 感想メモは「実際に〜してみたのですが」等の体験描写に展開する
- 数字・固有名詞はメモの記載をそのまま保持（勝手に変えない）
- メモに感想がないセクションでも、最低1文は所感を入れる
  → 書けない場合は `[★要加筆: ここに感想を追記]` とプレースホルダーを置く

変換例:

```
【メモ（入力）】
- Cherry MX互換、基板4層、KiCad設計
- 価格 ¥12,000（BOOTH販売あり）
- 感想: キーキャップの触り心地が独特

【文章化（出力）】
こちらはCherry MX互換のスイッチを採用した自作キーボードで、
基板は4層構成、KiCadで設計されているとのこと。
価格は12,000円で、BOOTHでも販売されています。
実際に触らせていただいたのですが、キーキャップの触り心地が
独特で、これは好みが分かれそうなところですね。
```

### プレースホルダーと加筆マーカー

テンプレートの `{{...}}` はNotionデータで自動置換する。
データが不足している場合は以下のマーカーを挿入し、人間の加筆を促す:

| マーカー | 意味 |
|---|---|
| `[★要加筆: 感想]` | メモに感想がなく、体験ベースの記述が必要 |
| `[★要加筆: 詳細]` | メモの情報が不足、現地の記憶で補完が必要 |
| `[★要確認: {{内容}}]` | 数字や固有名詞がメモから読み取れず確認が必要 |
| `[★写真: {{説明}}]` | 該当する写真がNAS内で見つからなかった |
| `[★アフィリエイト: {{種別}}]` | アフィリエイトリンクの挿入候補（人間が選定） |

WordPress下書きをエディタで開いた際、★マーカーを検索すれば加筆箇所が一覧できる。

### 記事タイプ別ルール

**聖地巡礼（Type A/B/C）:**
- Visits DB から Status = "Draft" を取得。作品別DB → 聖地スポットDB → 登場シーンDB のリレーションを辿る
- 構成: 挨拶+動機 → 【今回の聖地巡礼について】概要テーブル → 【スポット名】×N → 【グルメ・お土産】→ 【まとめ】→ 【参考・アクセス情報】
- 同じ旅行の複数記事には共通の Series tag を付与し、ナビカード用メニューに自動追加。
  各記事末尾に `[navi_list name="メニュー名"]` を埋め込み

**イベント（Type D）:**
- Events DB から ステータス = "素材整理完了" を取得
- イベントの正式名称を必ず1回はフルで記載
- 各見どころセクションは「概要 → 詳細 → 感想」の3段構成
- 講演内容は要約のみ（引用する場合は発言者名と文脈を明記）
- 技術イベントではスペック・型番を正確に記載（メモの数字をそのまま使う）
- 「次回も行きたい」「来年は○○に注目」など前向きな締めを心がける

**お出かけ（Type E）:**
- Events DB から ステータス = "素材整理完了" を取得
- アクセス情報を冒頭寄りに配置（読者がすぐ使える位置）
- 食事セクションは価格を必ず記載
- 季節・天候の影響がある場合は言及する
- 「○○がおすすめです」より「当方は○○が気に入りました」（押し付けない）

## パーマリンク・スラッグ命名規則

- パーマリンク構造: `/%postname%/`（Redirectionプラグインが変更を自動監視し301を作成）

| 記事タイプ | パターン | 例 |
|---|---|---|
| 聖地巡礼 | `{location}-pilgrimage-{year}-{season}` | `numazu-pilgrimage-2026-spring` |
| 技術記事 | `{topic-keyword}` | `unity-quaternion-rotation` |
| DIY | `diy-{project-name}` | `diy-led-matrix-controller` |
| イベント (Type D) | `event-{english-event-name}-{year}` | `event-makerfaire-tokyo-2026` |
| お出かけ (Type E) | `visit-{english-destination}-{year}` | `visit-kozushima-2026` |

メタディスクリプションは120文字以内で「場所/イベント名 + 何が見られるか + 一言感想」を含める。

## 著作権の注意点

**使えるもの（リスクが低い）:**
自分で撮影した現地の写真・動画 / 公式サイトへのリンク（URLのみ）/
アフィリエイト提供の商品画像 / 公式YouTubeの埋め込み（埋め込みコード使用）

**条件付き（引用の要件を厳密に守る）:**
アニメのスクリーンショットは1記事1〜2枚・最小サイズ・出所明記
（例: ©プロジェクトラブライブ!サンシャイン!!）。自分の文章がメイン、
画像は補足という関係を維持し、引用の必然性がある文脈でのみ使用。

**避けるべきもの:**
公式ロゴの転載 / 公式イラストのそのままの掲載 /
キャラクター画像をアイキャッチやヘッダーに使用。
